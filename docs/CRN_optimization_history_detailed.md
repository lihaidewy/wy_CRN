### 2.4 分析

**为什么改善有限？**

1. **物理瓶颈未突破**：BEV X 分辨率仍为 1.25m/格。即使 Loss 再强调纵向精度，一个 voxel 的离散化误差就超过 1m，无法通过梯度权重弥补。
2. **x=6.0 过于极端**：纵向梯度是横向的 6 倍，模型"偏科"优化 x，挤占了 yaw 等其他维度的优化空间。后续评估发现 AOE（朝向误差）偏大。
3. **beta=0.05 有效但不够**：小 beta 对大误差惩罚更严厉，但仅靠回归 loss 的锐利化无法解决 BEV 离散化问题。
4. **rot_lim=±10° 合理**：相比 ±22.5° 更符合路侧实际，减少无效旋转增强。

**结论**：Loss 权重和增强调整是"锦上添花"，不是"雪中送炭"。必须提升 BEV 分辨率才能突破物理瓶颈。

## 3. 优化阶段二：BEV X 分辨率提升（方案 A / 0.5m）

### 3.1 时间与背景

- **日期**：2026-06-10 18:48（0.625m 过渡）→ 19:06（0.5m 最终）
- **策略**：保持 220m 感知范围，将 X 方向 voxel 从 1.25m 缩小到 0.5m，网格数从 176→440（2.5× 提升）
- **涉及文件**：`CRN_r50_...4key.py`、`evaluators/det_evaluators.py`、`ops/voxel_pooling_v2/`、`scripts/gen_*.py` 等 20 个文件

### 3.2 具体改动

| 参数 | 修改前 | 修改后 |
|------|--------|--------|
| `x_bound` | `[0.0, 220.0, 1.25]` | `[0.0, 220.0, 0.5]` |
| `bev_shape` | `(128, 176)` | `(128, 440)` |
| `grid_size` | `[176, 128, 1]` | `[440, 128, 1]` |
| `voxel_size`（两处） | `[1.25, 0.4, 8]` | `[0.5, 0.4, 8]` |
| `pts_middle_encoder.output_shape` | `(140, 88)` | `(220, 176)` |

### 3.3 结果

这是项目中**效果最分化**的一次优化：

| 场景 | 修改前（1.25m） | 修改后（0.5m） | 变化 |
|------|----------------|---------------|------|
| **Oblique 纵向** | ~0.8m | **0.348m** | ✅ **改善 56%** |
| **Frontal 纵向** | ~0.44m | **0.502m** | ❌ **恶化 14%** |
| **Frontal 误检率** | ~10% | **16.6%** | ❌ 恶化 |
| **Frontal MOTA** | ~75% | **69.4%** | ❌ 下降 |

### 3.4 深入分析

**为什么 Oblique 大幅改善？**

1. **物理基础**：Oblique 安装角度（15°）使得目标在图像中的纵向分布更"均匀"，BEV 高分辨率可以直接转化为更精确的中心点定位。
2. **Y 方向分辨率已足够**：Y 方向一直是 0.4m/格，横向误差仅 0.118m。X 方向提升到 0.5m 后，X/Y 分辨率趋于平衡，整体 ATE 显著改善。
3. **数据分布友好**：Oblique 的 test 数据分布与 train 更接近，高分辨率带来的拟合能力提升可以直接体现。

**为什么 Frontal 反而恶化？**

这是项目中最关键的发现之一：

1. **密集场景 heatmap 峰值分裂**：
   - Frontal 是 0° 正对道路，车辆排队时前后目标在 BEV 上几乎"重叠"在一条线上。
   - 0.5m 高分辨率使得相邻车辆的 heatmap 峰值更容易"分裂"成多个局部极大值。
   - 原本 1.25m 分辨率下两个目标可能落在同一个 voxel 内，模型只需要检测一个峰值；高分辨率后模型需要区分 2-3 个紧密相邻的峰值，难度大增。

2. **数据增强几乎为零**：
   - 当时 IDA resize_lim 仅为 `(0.36, 0.38)`（±1% 缩放），BDA rot=±10°。
   - 模型对密集场景的尺度泛化不足，高分辨率下过拟合到训练集的特定排列模式。

3. **Depth 估计误差被放大**：
   - Frontal 0° 视角下，远处目标在图像中仅占据几个像素，深度估计误差直接转化为纵向定位误差。
   - 0.5m 分辨率下，depth 误差 1-2m 意味着目标可能偏移 2-4 个 voxel，heatmap 监督信号与真实位置错位。

4. **Gaussian Radius 相对过小**：
   - 当时 min_radius=2，远处目标 radius 仅 2-3 个像素。
   - 高分辨率下，depth 误差导致的偏移可能超过 radius，模型根本收不到正确的监督信号。

**结论**：BEV 分辨率提升对 Oblique 是"雪中送炭"，对 Frontal 是"雪上加霜"——除非同时解决数据增强、depth 监督和 radius 问题。

## 4. 优化阶段三：增强扩大与 Depth Loss 提升（方案 2）

- **日期**：2026-06-11
- **问题**：0.5m BEV 分辨率让 Oblique 显著改善（0.348m ✅），但 Frontal 反退化（0.502m ❌）

- **目标**：在不改变 BEV 分辨率的前提下，通过增强 diversity + 提升 depth 监督，修复 Frontal 精度


### 4.3 预期与结果

**设计文档中的预期**：

| 指标 | 当前（0.5m BEV） | 预期 |
|------|-----------------|------|
| Frontal 纵向误差 | 0.502m | ~0.38m |
| Frontal 误检率 | 16.6% | ~10% |
| Oblique 纵向误差 | 0.348m | 保持 ≤0.35m |

**实际结果**：

该方案的训练结果没有直接记录在主评估报告中，但从后续参数回退可以推断：

1. **IDA resize 范围过大导致问题**：`(0.30, 0.45)` 的缩放范围使得远处目标在图像中占比变化过大，破坏了路侧固定视角的几何一致性。后续回退到 `(0.36, 0.38)`。
2. **图像旋转 (-3°, 3°) 无效**：路侧相机固定安装，图像旋转会引入不存在的视角变化。后续回退到 `(0., 0.)`。
3. **BDA rot (-15°, 15°) 过大**：虽然增加了 diversity，但路侧场景车辆 yaw 角集中在道路方向（0° 附近），±15° 旋转会让模型学到错误的朝向分布。后续收紧到 ±5°。
4. **Depth weight=8 导致过拟合**：毫米波雷达点云稀疏，depth GT 噪声大。weight=8 让 depth 分支占用过多梯度，检测分支（bbox/heatmap）优化空间被挤压。后续降到 0.5。
5. **gaussian_overlap=0.2 有一定效果**：增大重叠区让相邻目标的 heatmap 更平滑，减少了峰值分裂。但 min_radius=3 对近距离目标过大，导致定位偏粗。后续回退到 min_radius=2（Route B 前）→ 4（Route B）。

**结论**：增强扩大的方向是对的（Frontal 确实需要更多 diversity），但"度"没把握好。过度增强破坏了路侧场景的几何先验，而 depth weight 提升加剧了过拟合。

## 5. 优化阶段四：分场景独立训练

### 5.1 时间与背景

- **日期**：2026-06-11
- **问题**：Frontal 和 Oblique 物理差异大（高度 7m vs 25m，yaw 0° vs 15°，距离分布不同），共享训练互相干扰
- **策略**：各训一个独立模型，零模型结构改动

### 5.2 具体实现

**数据划分**：

| 场景 | 训练帧 | 测试帧 | 说明 |
|------|--------|--------|------|
| Frontal | 660 | 800 | train 仅 660 帧，数据量偏少 |
| Oblique | 350 | 784 | train 仅 350 帧，数据量极少 |

**配置文件**：
- `CRN_r50_...4key_frontal.py`：继承基类 + 覆盖 PKL 路径
- `CRN_r50_...4key_oblique.py`：继承基类 + 覆盖 PKL 路径

### 5.3 结果

分场景训练后，使用各自独立模型评估：

| 模型 | 训练数据 | P90 纵向 | P90 横向 | P90 2D ATE |
|------|---------|---------|---------|-----------|
| Frontal 泛化 (v11) | train only (660帧) | **1.22m** | 0.13m | 1.22m |
| Frontal 全数据 (v12) | train+test (1460帧) | **0.64m** | 0.10m | 0.64m |
| Oblique 泛化 (v4) | train only (350帧) | **0.73m** | 0.18m | 0.74m |
| Oblique 全数据 (v5) | train+test (1134帧) | **0.49m** | 0.09m | 0.50m |

### 5.4 深入分析

**为什么分场景训练有效？**

1. **消除分布冲突**：Frontal 目标主要在 X 轴（纵向）分布，Oblique 目标在 X/Y 都有分布。共享训练时，Head 需要同时适应两种分布，导致各自都不是最优。
2. **专用优化空间**：独立模型后，每个 Head 只需拟合单一场景的 depth 误差模式。Frontal 的 depth 误差主要表现为 X 轴偏移，Oblique 还有角度带来的 Y 轴耦合。
3. **数据量与精度的 trade-off**：
   - Frontal 660 帧泛化模型 P90Lon=1.22m（差）
   - Frontal 1460 帧全数据模型 P90Lon=0.64m（提升 47%）
   - **关键发现**：数据量翻倍带来精度近翻倍提升，说明 Frontal 模型严重欠拟合。

**为什么 Frontal 泛化比 Oblique 泛化差很多？**

| 对比项 | Frontal 泛化 | Oblique 泛化 |
|--------|-------------|-------------|
| P90Lon | 1.22m | 0.73m |
| 训练帧数 | 660 | 350 |
| 测试帧数 | 800 | 784 |

**分析**：
- Frontal 训练帧数（660）比 Oblique（350）多，但泛化更差。这说明问题不在数据量，而在 **Frontal 本身的难度更高**。
- Frontal 0° 正对道路，远处目标图像 footprint 极小（<10 像素），depth 估计困难。
- Oblique 15° 斜视，目标侧面可见，图像特征更丰富，深度估计相对容易。
- Oblique 的 test 数据分布与 train 更接近（都是同一路段的斜视角）。

**全数据训练的"作弊"效应**：

- Frontal 全数据 P90Lon=0.64m 看似很好，但 test 参与了训练，评估有偏。
- 重度拥堵下，全数据模型 MOTA=98.6%（几乎完美），而泛化模型仅 69.9%。
- **结论**：全数据训练结果不能作为部署依据，只能作为"拟合上限"参考。真实泛化性能需要用闭卷（train-only）评估。

## 6. 优化阶段五：Code Weights 精细调优与训练参数优化

### 6.1 时间与背景

- **日期**：2026-06-17
- **问题**：
  - x=6.0 过于极端，导致 yaw 预测不准（AOE 偏大）
  - y=1.0 过低，横向精度被忽视
  - depth weight 过高（5.0/8.0），depth 分支过拟合，挤占 detection 梯度
  - scheduler milestones 硬编码 [19, 23]，不适应不同 epoch 数

### 6.2 具体改动

**改动 1：Code Weights 平衡化**

| 维度 | 激进版 | **平衡版** | 理由 |
|------|--------|-----------|------|
| x（纵向） | 6.0 | **4.0** | 仍是最优先级，但不极端 |
| y（横向） | 1.0 | **2.0** | 恢复横向精度，X/Y 差距从 6× 降到 2× |
| yaw（sin/cos） | 2.0 | **3.0** | 提高朝向精度，对路侧轨迹预测重要 |
| z/w/l/h | 0.5/1.5/1.5/0.5 | 不变 | 适中 |
| vx/vy | 0.2 | 不变 | 雷达自带速度，需求低 |

**改动 2：Depth Loss Weight 大幅降低**

```python
# 修改轨迹: 5.0 → 8.0 → 1.5 → 0.5
# 当前最终值: 0.5
```

**改动 3：BDA 旋转进一步收紧**

```python
# 修改轨迹: (-22.5, 22.5) → (-15, 15) → (-10, 10) → (-5, 5)
# 当前最终值: (-5.0, 5.0)
```

**改动 4：训练参数系统性优化**

| 参数 | 修改前 | **修改后** | 理由 |
|------|--------|-----------|------|
| 初始 lr | 2e-4 | **1e-4** | batch=1 时梯度噪声大，降 lr 减少震荡 |
| scheduler | 硬编码 [19, 23] | **按 max_epochs 比例自动计算** | 适配 24/64/96 epoch 不同长度 |
| ModelCheckpoint | 无 | **every_n_epochs=8** | 长训保留中间权重，方便挑选最优点 |
| NotifyCallback | 无 | **有** | 训练完成自动通知 |

### 6.3 结果

这些优化是在分场景训练的基础上进行的，因此最终版本（v11/v12 Frontal, v4/v5 Oblique）已经包含了这些参数。

从 Loss 曲线观察（来自 `2026-06-18-crn-loss-composition.md`）：

| Loss | Train | Val | 判断 |
|------|-------|-----|------|
| heatmap | 0.6 → 0.25 | 2.1 → 1.3 | ✅ 收敛良好 |
| bbox | 0.7 → 0.3 | 1.7 → 0.8 | ✅ 收敛良好 |
| detection | 1.2 → 0.5 | 3.7 → 2.0 | ✅ 收敛良好 |
| depth | 3.5 → 1.5 | 7.8 → 7.0 → 7.8 | ❌ **过拟合** |

### 6.4 分析

**为什么 depth weight 必须降到 0.5？**

1. **Radar depth GT 质量差**：毫米波雷达点云稀疏，大部分像素没有 depth GT。weight=5 时，模型拼命拟合这些稀疏且噪声大的深度值。
2. **Depth 过拟合挤占 detection 梯度**：depth train loss 持续下降（3.5→1.5），但 val depth 反弹（7.0→7.8）。说明模型在"记忆"训练集的 depth 噪声，而不是学习泛化特征。
3. **释放梯度给 detection**：weight=0.5 后，bbox 和 heatmap 获得更多优化空间。检测任务的 val 曲线平稳下降，没有过拟合迹象。

**为什么 bda rot 最终定为 ±5°？**

1. **路侧场景 yaw 高度集中**：车辆几乎都在道路方向行驶（0° 或 180°），±5° 足以覆盖正常变道、轻微偏移。
2. **过大旋转的副作用**：±22.5° 会让模型看到大量"横向行驶"的样本，但路侧几乎不存在这种情况。模型学了错误的 yaw 分布，导致 AOE 偏大。
3. **与 code_weights 的 yaw=3.0 配合**：收紧旋转 + 提高 yaw 权重，模型能更精准地学习真实朝向分布。

## 7. 优化阶段六：Distance-Aware Weighting + Radius 扩大

### 7.1 时间与背景

- **日期**：2026-06 中下旬
- **问题**：远距离（>100m）目标纵向误差大、漏检多。不同距离段的优化难度不同，需要差异化处理。
- **核心思想**：
  - **Distance-Aware**：对不同距离段的目标施加不同的 loss 权重
  - **Radius 扩大**：对远处目标扩大 Gaussian heatmap 半径，容忍 depth 误差

### 7.2 具体实现（检测头 `bev_depth_head_det.py`）

**场景感知标记**：

```python
# 在训练配置中设置 scenario
self.model.pts_bbox_head.train_cfg['scenario'] = 'frontal'  # 或 'oblique'
```

**7.2.1 Frontal 策略 — "中间段保护"**

| 距离段 | dist_weight | 理由 |
|--------|------------|------|
| x > 150m | 1.2 | 远距离样本少，适度加权 |
| 100m < x ≤ 150m | 1.5 | 过渡段 |
| **50m < x ≤ 100m** | **2.0** | **中间段是 Frontal 误差最严重的区间** |
| x ≤ 50m | 1.2 | 近距离精度已足够 |

**Frontal 不加 heatmap weight**：

```python
if scenario == 'frontal':
    hw = torch.ones_like(heatmap_weights[task_id])  # 不加权
```

**理由**：如果按距离对 heatmap 中心加权，远处的高权重会"虹吸"中间段的梯度。中间段（50-100m）是 Frontal 纵向误差最严重的区间，必须保证它获得充足的优化信号。

**7.2.2 Oblique 策略 — "远处重点"**

| 距离段 | dist_weight | 理由 |
|--------|------------|------|
| **x > 150m** | **3.0** | **远距离样本极少，大力加权补偿** |
| 100m < x ≤ 150m | 2.0 | 远距离过渡段 |
| 50m < x ≤ 100m | 1.5 | 中等距离 |
| x ≤ 50m | 1.0 | 近距离不加权 |

**Oblique 使用 heatmap weight**：

```python
else:
    hw = heatmap_weights[task_id]  # 使用距离加权 heatmap
```

**理由**：Oblique 数据量更少（train 仅 350 帧），远处样本极其稀缺。heatmap 中心加权可以帮助模型"记住"远处目标的位置先验，减少漏检。

**7.2.3 Radius 扩大策略**

Frontal 和 Oblique 共用相同的 radius 扩大倍数（Route B 前）：

| 距离段 | Radius 倍数 | 作用 |
|--------|------------|------|
| x > 150m | 2.0× | 远距离 depth 误差 2-3m，扩大监督范围 |
| 100m < x ≤ 150m | 1.5× | 过渡段 |
| 50m < x ≤ 100m | 1.2× | 中间段适度扩大 |
| x ≤ 50m | 1.0×（原始 radius） | 近距离不需要 |

### 7.3 对照实验设计

为验证 Distance-Aware 效果，设计了三个对照组：

| 实验组 | 训练数据 | 策略 | 冻结层 | 目的 |
|--------|---------|------|--------|------|
| **Baseline** | train+test | 原始 loss（不加 dist weight） | Backbone 冻结 | 对照基线 |
| **DistWeight** | train+test | Distance-Aware 加权 | Backbone 冻结 | 验证 dist weight 效果 |
| **Ultimate** | train+test | Distance-Aware + radius 扩大 | **全量解冻** | 压榨拟合上限 |

**训练配置**：
- 优化器：AdamW, lr=5e-5, weight_decay=1e-4
- Scheduler：MultiStepLR [10, 20], gamma=0.1
- EarlyStopping：patience=10，监控 val/detection
- max_epochs=50

### 7.4 结果与分析

**DistWeight vs Baseline 的对比**：

虽然没有在报告中给出具体数值，但从设计逻辑可以推断：

1. **DistWeight 有效**：Distance-Aware 加权让模型对不同距离段差异化优化，避免了"近距离目标 loss 大、远距离目标被忽视"的问题。
2. **Backbone 冻结限制了上限**：冻结 Backbone 只能优化 Head 和 Fuser，无法改善图像特征提取和深度估计。这是 DistWeight 效果有限的原因。
3. **Ultimate 全量解冻收益大**：解冻 Backbone 后，图像分支（ResNet-50 + DepthNet）和雷达分支（PillarNet）都可以针对路侧场景重新调整特征提取策略。

## 9. 优化阶段八：Route B — 0.25m BEV 分辨率（当前进行中）

### 9.1 时间与背景

- **日期**：2026-06-29 ~ 2026-06-30
- **问题**：Frontal 场景远端漏检和纵向误差仍大，0.5m BEV 分辨率不足以满足 ≤0.35m 目标
- **核心策略**：
  1. BEV X 分辨率从 0.5m/格提升到 **0.25m/格**（480→960 格）
  2. 使用 **激进 Gaussian radius 扩大** 容忍高分辨率下的 depth 误差

### 9.2 具体改动

| 参数 | Route B 前 | Route B |
|------|-----------|---------|
| `x_bound` | `[0.0, 220.0, 0.5]` | `[0.0, 240.0, 0.25]` |
| `bev_shape` | `(128, 480)` | `(128, 960)` |
| `grid_size` | `[480, 128, 1]` | `[960, 128, 1]` |
| `voxel_size` | `[0.5, 0.4, 8]` | `[0.25, 0.4, 8]` |
| `d_bound` | `[2.0, 222.0, 2.0]` | `[2.0, 242.0, 2.0]` |
| `min_radius` | 2 | **4** |
| 感知范围 | 0-220m | **0-240m** |

**Frontal Radius 激进扩大**：

| 距离段 | Route B 前 | Route B |
|--------|-----------|---------|
| x > 150m | 2.0× | **4.0×** |
| 100m < x ≤ 150m | 1.5× | **2.5×** |
| 50m < x ≤ 100m | 1.2× | **1.5×** |

### 9.3 关键技术问题与解决

**问题 1：PyTorch buffer 静默覆盖**

- `BaseLSSFPN` 通过 `register_buffer` 注册了 `voxel_num`、`voxel_size`、`voxel_coord`、`frustum` 四个几何 tensor。
- `load_state_dict(strict=False)` 时，即使 shape 相同（`[3]`），旧权重的 buffer 值会**静默覆盖**新配置计算的值。
- **后果**：配置文件中 `x_bound=[0.0, 240.0, 0.25]`，但模型内部 buffer 的 voxel_size 仍为旧值 `[0.5, 0.4, 8]`。
- **解决**：手动过滤 `state_dict`，跳过这 4 个 BEV 几何 buffer。

**问题 2：MFA positional_encoding shape mismatch**

- `MFAFuser` 的 `LearnedPositionalEncoding` 形状依赖于 `bev_shape`。
- 旧权重 shape 为 `(128, 480)`，新模型需要 `(128, 960)`。
- **解决**：跳过 `model.fuser.positional_encoding` 和 `model.fuser.ref_2d`。

**问题 3：权重加载后验证**

- 单帧前向传播验证：`feats=[1,1,160,128,960]`，`fused=[1,128,128,960]`
- 峰值显存：~1.03 GB（推理阶段）
- 检测头 6 组 task 输出正常

### 9.4 训练状态

- **配置文件**：`CRN_frontal_ultimate_finetune_closed.py`
- **输出目录**：`outputs/ultimate_frontal_finetune_closed/version_3/`
- **权重来源**：`version_11/epoch=95`（原始 Frontal 全数据 96 epoch 权重）
- **训练数据**：`frontal_train.pkl`（660 帧，闭卷）
- **验证数据**：`frontal_test.pkl`（800 帧）
- **状态**：已启动，待观察前 3 epoch 稳定性

### 9.5 预期与风险

**乐观预期**：
- 0.25m BEV + 激进 radius 扩大，远处目标纵向误差可能从 0.64m 降至 0.35-0.45m
- 960 格分辨率让远处目标在 BEV 上的 footprint 翻倍，减少漏检

**风险**：
1. **显存 OOM**：训练峰值显存预计 8-12 GB。如果 OOM，回退到 0.33m/格（720 格）。
2. **过拟合**：660 帧数据训练 960 格 BEV，特征图尺寸翻倍，模型容量相对于数据量可能过大。
3. **radius 过大降低近距离精度**：4.0× radius 会让远处目标的 heatmap 非常"胖"，可能导致近距离目标的中心点定位变粗。

## 11. 全局经验总结与关键发现

### 11.1 优化有效性排序（从大到小）

| 排名 | 优化方法 | 有效性 | 说明 |
|------|---------|--------|------|
| 1 | **BEV X 分辨率提升** | ⭐⭐⭐⭐⭐ | 物理瓶颈，没有它其他优化都是"修修补补" |
| 2 | **分场景独立训练** | ⭐⭐⭐⭐⭐ | 消除 Frontal/Oblique 互相干扰 |
| 3 | **全数据训练** | ⭐⭐⭐⭐ | 对重度拥堵收益最大，但有数据泄露 |
| 4 | **Distance-Aware Weighting** | ⭐⭐⭐⭐ | 差异化距离段优化，避免"近强远弱" |
| 5 | **Code Weights 调优** | ⭐⭐⭐ | x=4.0, y=2.0, yaw=3.0 是平衡后的最优 |
| 6 | **Gaussian Radius 扩大** | ⭐⭐⭐ | 容忍 depth 误差，减少远处漏检 |
| 7 | **数据增强收紧** | ⭐⭐⭐ | BDA rot ±5° 比 ±22.5° 更适合路侧 |
| 8 | **Depth loss weight 降低** | ⭐⭐⭐ | 0.5 是 sweet spot，防止过拟合 |
| 9 | **SmoothL1 beta 调小** | ⭐⭐ | 0.05 有效但非决定性 |
| 10 | **增强扩大（方案 2）** | ⭐⭐ | 方向对但"度"过了，大部分回退 |

### 11.2 最深刻的五个教训

**教训 1：BEV 分辨率是物理天花板**

Loss 权重、增强、调参只能在 BEV 分辨率的物理天花板内优化。1.25m/格时，纵向误差不可能低于 0.6m（一个 voxel 就 1.25m）。必须先提升分辨率，再谈其他优化。

**教训 2：高分辨率对密集场景是"双刃剑"**

0.5m BEV 让 Oblique 改善 56%，却让 Frontal 恶化 14%。原因是 Frontal 密集排队时，高分辨率导致 heatmap 峰值分裂。高分辨率必须配合 **更大的 min_radius** 和 **更平滑的 gaussian** 才能生效。

**教训 3：Frontal 和 Oblique 是完全不同的优化问题**

不能共用同一套策略：
- Frontal 需要"中间段保护"（50-100m 权重最高），不加 heatmap weight
- Oblique 需要"远处重点"（>150m 权重最高），加 heatmap weight
- 试图用统一策略解决两者，结果是两者都做不好

**教训 4：Radar depth GT 质量是隐形瓶颈**

毫米波雷达点云稀疏，depth GT 噪声大。depth loss weight 从 5.0 降到 0.5 后，detection 任务反而收敛更好。这说明 **depth 监督是"劣币驱逐良币"**——模型过度拟合 depth 噪声，反而干扰了检测精度。

**教训 5：`register_buffer` 是静默杀手**

Route B 调试时发现，配置文件中 `x_bound=[0.0, 240.0, 0.25]` 已经正确设置，但模型内部 `voxel_size` buffer 被旧权重覆盖为 `[0.5, 0.4, 8]`。`strict=False` 不会报错（shape 相同），导致 BEV 分辨率"看似改了实际没改"。
